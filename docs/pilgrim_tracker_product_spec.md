# Pilgrim Tracker Product Specification

## 1. Product Overview

### 1.1 Product Name

The application name is **Pilgrim Tracker**.

Build **Pilgrim Tracker**, a cross-platform, local-first expense, asset, and finance tracking application for personal use, projects, and businesses. The look of the app should be PROFESIONAL AND PREMIUM.

The application must support:

- Income tracking
- Expense tracking
- Asset transfers
- Asset conversions between assets with different units, such as IDR cash into grams of gold
- Quantity-based asset tracking for gold, stocks, cryptocurrency, inventory, and other measurable assets
- Project and business financial tracking
- Recurring transactions
- Tithe obligation tracking
- Fast transaction entry
- Desktop and mobile editing
- Offline operation
- Online synchronization
- Import and export
- Receipt and bill scanning in a later phase

The initial target platforms are:

- Android
- Windows desktop

The architecture should remain extensible for:

- iOS
- macOS
- Linux
- Web

---

## 2. Core Product Principles

### 2.1 Local-first

The application must continue working without internet access.

Users must be able to:

- Create transactions
- Edit transactions
- Delete or archive transactions
- View balances
- View reports
- Manage projects
- Manage accounts
- Manage tithe obligations
- Create recurring transactions
- Import and export supported files

All changes are stored locally first and synchronized later when internet access is available.

### 2.2 Mobile and desktop are equal clients

The phone application is optimized for fast data entry.

The desktop application is optimized for:

- Detailed editing
- Spreadsheet-like transaction management
- Bulk editing
- Reporting
- Import and export
- Reconciliation

Both clients must be able to create, edit, delete, restore, import, export, and synchronize financial records.

### 2.3 Simple user interface, reliable accounting model

The interface should feel like a normal expense manager.

Internally, use a lightweight double-entry ledger so that:

- Transfers do not count as income or expenses
- Account balances remain reliable
- Reports are consistent
- Future accounting features can be added safely

### Client applications

Use Flutter for:

- Android

- Future iOS, macOS, and Linux support

### Local database

Use SQLite.

Recommended Flutter packages may include:

- drift
- sqlite3_flutter_libs
- riverpod or bloc
- freezed
- go_router

### Server

Recommended backend:

- PostgreSQL
- REST or GraphQL synchronization API
- Object storage for receipt images
- Token-based authentication
- Encrypted backups

Possible implementation options:

- Supabase
- Custom backend using FastAPI, NestJS, or ASP.NET Core

### Money storage

Store monetary values as integers in the smallest currency unit.

Examples:

- 10.25 USD is stored as 1025
- 150,000 IDR is stored as 150000

Do not use floating-point values for money.

## 5. Transaction Types

The system must support:

1. Income
2. Expense
3. Transfer
4. Asset conversion
5. Asset sale or disposal
6. Tithe payment
7. Refund
8. Balance adjustment
9. Split transaction
10. Recurring transaction instance

### 5.1 Income

An income transaction increases an asset account.

Example:

- Income category: Sales
- Destination account: Business Bank
- Amount: 2,000,000

### 5.2 Expense

An expense transaction reduces an asset account.

Example:

- Source account: Business Bank
- Expense category: Office Supplies
- Amount: 250,000

### 5.3 Transfer

A transfer moves money between accounts that use the same or compatible unit.

Example:

- From: Bank IDR
- To: Cash IDR
- Amount: 500,000

Transfers must not increase income or expenses.

### 5.4 Asset conversion

An asset conversion exchanges one asset for another asset with a different measurement unit.

Examples:

- IDR cash into grams of gold
- Cash into shares
- Cash into cryptocurrency units
- Cash into inventory
- One commodity into another commodity

Example:

```text
Transaction type: Asset Conversion
From account: Business Bank
From amount: IDR 50,000,000
To asset account: Gold Holdings
Quantity received: 20
Unit: gram
Acquisition unit price: IDR 2,500,000 per gram
Total acquisition cost: IDR 50,000,000
```

This transaction must:

- Reduce the source asset
- Increase the destination asset quantity
- Preserve the acquisition cost or cost basis
- Not count as income
- Not count as an ordinary operating expense
- Preserve total asset value at acquisition, excluding separately expensed fees

Internal accounting entry:

| Ledger entry |          Debit |         Credit |
| ------------ | -------------: | -------------: |
| Gold asset   | IDR 50,000,000 |              — |
| Cash or bank |              — | IDR 50,000,000 |

The destination asset account stores:

```text
Quantity: 20 grams
Total cost basis: IDR 50,000,000
Average unit cost: IDR 2,500,000 per gram
```

#### Asset conversion fees

The user can choose one of two treatments:

1. Capitalize the fee into the asset cost basis
2. Record the fee as a separate expense

Example:

```text
Gold purchase price: IDR 50,000,000
Administration fee: IDR 200,000
Total cash paid: IDR 50,200,000
```

Capitalized treatment:

```text
Gold cost basis: IDR 50,200,000
Quantity: 20 grams
Average unit cost: IDR 2,510,000 per gram
```

Expense treatment:

```text
Gold asset: IDR 50,000,000
Administration expense: IDR 200,000
Cash reduction: IDR 50,200,000
```

#### Asset valuation

A quantity-based asset account may store a market price separately from its historical cost.

Example:

```text
Quantity: 20 grams
Current market price: IDR 2,700,000 per gram
Current market value: IDR 54,000,000
Historical cost basis: IDR 50,000,000
Unrealized gain: IDR 4,000,000
```

Unrealized gains and losses must not be treated as normal cash income or expense.

#### Partial asset sale or disposal

When part or all of an asset is sold, Pilgrim Tracker calculates the removed cost basis and realized gain or loss.

Example:

```text
Gold sold: 5 grams
Sale price: IDR 3,000,000 per gram
Cash received: IDR 15,000,000
Removed cost basis: IDR 12,500,000
Realized gain: IDR 2,500,000
Remaining quantity: 15 grams
```

The first implementation should use weighted-average cost basis.

Buying or converting into an asset must not create a new tithe obligation by itself. Tithe normally originates from the eligible income that produced the cash used for the conversion.

### 5.5 Asset sale or disposal

An asset sale converts part or all of a quantity-based asset back into cash or another asset.

The transaction must record:

- Quantity sold
- Unit
- Sale price per unit
- Gross proceeds
- Fees
- Removed cost basis
- Realized gain or loss
- Remaining quantity
- Remaining cost basis

### 5.6 Split transaction

One transaction may contain multiple categories or projects.

Example:

Total transaction: 1,000,000

- Office Supplies: 600,000
- Transport: 250,000
- Food: 150,000

## 10. Tithe Tracking

The tithe feature must be implemented using three separate entities:

1. Tithe rule
2. Tithe obligation
3. Tithe payment allocation

### 10.1 Tithe rule

Example rule:

```text
Rate: 13%
Effective from: 2026-01-01
Basis: Gross eligible income
Book: Main Business
Categories: Sales, Service Income
```

When the rate changes:

```text
Rate: 14%
Effective from: 2026-02-01
```

Do not edit the historical 13% rule.

Create a new version.

### 10.2 Tithe rate snapshot

Every eligible income transaction must store the applicable tithe rate.

Example:

```text
Income amount: 1,000,000
Rate snapshot: 13%
Tithe due: 130,000
Rule ID: tithe-rule-2026-01
```

Changing the current rate to 14% must not change old transactions automatically.

### 10.3 Tithe obligation

Recommended fields:

```text
id
income_transaction_id
rule_id
period
eligible_amount
rate_snapshot
amount_due
amount_allocated
status
created_at
updated_at
```

Statuses:

- Unpaid
- Partially paid
- Paid
- Overpaid
- Waived

### 10.4 Tithe payment

A tithe payment is a financial transaction.

The payment can be allocated to:

- Oldest outstanding obligation
- Selected month
- Selected income transactions
- Advance tithe balance

### 10.5 Example

January 2026:

```text
Eligible income: 10,000
Rate: 13%
Required tithe: 1,300
Paid: 1,000
Outstanding: 300
```

February 2026:

```text
Eligible income: 5,000
Rate: 14%
Required tithe: 700
Paid: 500
Outstanding: 200
```

Total outstanding:

```text
500
```

### 10.6 Editing eligible income

If an income amount changes:

```text
Original income: 1,000,000
Original rate: 13%
Original tithe due: 130,000

Updated income: 1,200,000
Updated tithe due: 156,000
Additional amount due: 26,000
```

Show a confirmation before recalculating.

### 10.7 Editing income date

A changed date may use a different tithe rule.

Example:

```text
2026-01-31 uses 13%
2026-02-01 uses 14%
```

The application should display:

```text
The corrected date uses a different tithe rule.

Previous rate: 13%
New rate: 14%
```

Recommended default:

- Use the rule applicable to the corrected transaction date
- Record the recalculation in transaction history

### 10.8 Overpayment handling

If a tithe payment exceeds the obligation, allow the user to classify the difference as:

- Advance tithe
- Tithe credit
- Voluntary additional giving

## 12. Offline Synchronization

### 12.1 Local-first workflow

When a transaction is created offline:

1. Save it immediately to SQLite
2. Mark it as pending sync
3. Add it to the local change queue
4. Upload when internet access returns
5. Mark as synchronized after server confirmation

### 12.2 Synchronization metadata

Each synchronized entity should include:

```text
id
version
created_at
updated_at
deleted_at
device_id
sync_status
last_synced_at
```

Recommended sync statuses:

- Local only
- Pending
- Syncing
- Synced
- Conflict
- Failed

### 12.3 Global identifiers

Use UUID, ULID, or another globally unique identifier.

Do not use only auto-incrementing local IDs.

### 12.4 Conflict handling

Safe automatic merges:

- Description changed on one device
- Tags changed on another device
- Notes changed independently

Manual conflict review required for:

- Amount
- Currency
- Date
- Source account
- Destination account
- Tithe rate
- Tithe amount
- Deleted versus edited records

Conflict interface example:

| Field  |  Original |     Phone |   Desktop |
| ------ | --------: | --------: | --------: |
| Amount | 1,000,000 | 1,250,000 | 1,500,000 |

Options:

- Keep phone version
- Keep desktop version
- Enter another value
- Preserve both as separate transactions

### 12.5 Deletions

Use tombstones:

```text
deleted_at: 2026-07-18T10:30:00Z
```

Do not immediately hard-delete synchronized data.

### 12.6 Attachment synchronization

Receipt images should synchronize separately.

Requirements:

- Compress images
- Calculate file hash
- Prevent duplicate uploads
- Retry failed uploads
- Keep transaction usable even when attachment upload fails
- Show attachment sync status

## 21. Important Acceptance Criteria

### Offline behavior

- The user can create and edit transactions without internet.
- Transactions appear immediately.
- Reports update immediately.
- Changes synchronize after the connection returns.

### Transfer behavior

- A transfer changes account balances.
- A transfer does not increase income.
- A transfer does not increase expenses.

### Asset conversion behavior

- IDR 50,000,000 can be converted into 20 grams of gold.
- Cash decreases by IDR 50,000,000.
- Gold quantity increases by 20 grams.
- Gold cost basis increases by IDR 50,000,000.
- Average unit cost becomes IDR 2,500,000 per gram.
- The conversion does not count as income.
- The conversion does not count as an ordinary expense.
- Conversion fees can be capitalized or expensed.
- Market-value changes are tracked separately as unrealized gains or losses.
- A partial sale reduces quantity and cost basis correctly.
- Realized gain or loss is calculated when an asset is sold.
- An asset conversion does not independently create a tithe obligation.

### Tithe behavior

- Income before a rule change keeps the old rate.
- Income after the rule change uses the new rate.
- Every eligible income stores a rate snapshot.
- Partial tithe payments are tracked.
- Outstanding tithe can be viewed by month.
- Overpayments can become advance tithe.
- Editing income recalculates obligations with confirmation.
- Changing an income date can change the applicable rule with confirmation.

### Editing behavior

- Transactions can be edited on phone and desktop.
- Every important edit creates a revision.
- Deleted records can be restored.
- Conflicting amount edits are not silently overwritten.

### Import behavior

- The user can preview imported records.
- Invalid rows are identified.
- Duplicate candidates are identified.
- A complete import batch can be rolled back.

### Export behavior

- CSV and XLSX files open correctly in spreadsheet software.
- Full JSON backup preserves relationships.
- A backup can be restored into a clean installation.

## 23. Final Product Definition

Pilgrim Tracker is a local-first, cross-platform expense, asset, and project finance manager with:

- Android and Windows applications
- Fast transaction entry
- Spreadsheet-like desktop editing
- Reliable account transfers
- Asset conversions across different units
- Quantity-based asset tracking
- Historical cost, market value, and realized or unrealized gain tracking
- Project and business reporting
- Versioned tithe rules
- Tithe obligation and payment tracking
- Offline operation
- Delayed online synchronization
- Import, export, backup, and restore
- Future receipt scanning

The desktop application acts as the financial control center.

The phone application acts as the fast transaction capture tool.

Both applications are complete financial clients and use the same synchronized data model.
